# Технический дизайн: Этап 1 (запуск, онбординг, экран «Мои слова»)

- **Дата:** 2026-06-27
- **Статус:** на согласовании
- **Базовый дизайн:** [2026-06-27-irregular-verbs-bot-design.md](2026-06-27-irregular-verbs-bot-design.md)
  (архитектура пакетов уточнена ниже — ports & adapters)
- **Источник требований:** [README.md](../../../README.md)

## 1. Цель этапа

Тонкий вертикальный срез MVP. После этого этапа бот:

1. **Запускается** — читает конфиг, подключается к MongoDB, сидит слова из
   `data/verbs.json`, крутит цикл long polling (`GetUpdates`).
2. **Онбордит пользователя** — по `/start` пользователь выбирает уровень,
   вариант форм (GB/US) и порядок изучения; профиль сохраняется в Mongo.
3. **Показывает экран «Мои слова»** — пока без обучения, поэтому экран в
   состоянии «пусто» (заглушка с текстом).

Этот этап — подмножество MVP. Логика обучения, напоминания и отбор слов в этот
этап **не входят** и будут добавлены позже без переделки уже написанного.

## 2. Архитектура: ports & adapters

Бизнес-логика — в центре (`service`), вокруг — тонкие пакеты-адаптеры под
каждую внешнюю зависимость. **Правило зависимостей:** всё указывает внутрь.
`service` объявляет интерфейсы-порты (`UserRepository`, `VerbRepository`),
а адаптеры их реализуют. `service` **не импортирует** ни `store`, ни `bot`,
поэтому покрывается unit-тестами без сети и БД.

```
            +----------------------+
            |   internal/service   |  бизнес-логика + типы + порты + FSM
            +----------------------+
              ^         ^         ^
   реализует  |         | импорт  | импорт
   порты      |         |         |
   +----------+--+   +--+-------+  (reminder — этап 5)
   | internal/   |   | internal/|
   | store       |   | bot      |
   | (Mongo)     |   |(Telegram)|
   +-------------+   +----------+
            ^             ^
            |             |
          +-------------------+
          |     cmd/bot       |  composition root (+ internal/config)
          +-------------------+
```

| Пакет | Слой | Ответственность | Зависит от |
|-------|------|-----------------|------------|
| `cmd/bot` | entry | Composition root: собрать всё, запустить polling, graceful shutdown. | все |
| `internal/config` | adapter | Чтение env: `BOT_TOKEN`, `MONGO_URI`, `MONGO_DB`. | — |
| `internal/service` | core | Вся бизнес-логика: типы (`Verb`, `User`, `Settings`, `State`), порты (`UserRepository`, `VerbRepository`), FSM-экраны (`Screen`), use-cases онбординга и меню, загрузка/сидинг слов. | — |
| `internal/store` | adapter | MongoDB: реализует порты `service`, хранит users и verbs. | `service` |
| `internal/bot` | adapter | Telegram: poll-роутер, маппинг апдейтов в вызовы `service`, рендер экранов и клавиатур. | `service` |

**Не входит в этап 1:** `internal/reminder` (этап 5), порт `Notifier`.

## 3. Модель данных

### Коллекция `verbs`

Сид из `data/verbs.json` при старте, upsert по `_id = base`. Все 170 слов.

### Коллекция `users` — агрегат пользователя

`_id = telegram user id`. **Единственный, кто пишет этот документ, —
`service`.** `bot` читает состояние, чтобы интерпретировать ввод, но не
пишет. На этом этапе храним профиль и состояние FSM; карта `words`
добавится вместе с обучением.

```jsonc
{
  "_id": 123456789,
  "settings": { "level": "elementary", "variant": "gb", "order": "alpha" },
  "state":    { "screen": "main_menu" },
  "created_at": "...", "last_active_at": "..."
}
```

Состояние FSM (`state.screen`) хранится в Mongo → переживает перезапуск.

## 4. FSM и use-cases (в `service`)

Переходы экранов — это бизнес-логика, поэтому FSM живёт в `service`. Каждый
use-case загружает пользователя, меняет состояние, сохраняет и возвращает
**следующий экран** (`Screen`). `bot` рендерит экран в текст и клавиатуру.

Экраны:

```
onboarding_level → onboarding_variant → onboarding_order → main_menu → my_words
```

Методы `service` для этапа 1 (возвращают `Screen, error`):

| Метод | Логика |
|-------|--------|
| `Start(userID)` | Нет пользователя → создать, экран `onboarding_level`. Уже онбордился → `main_menu` (профиль не трогаем). |
| `SetLevel(userID, level)` | Валидирует `level`, сохраняет, экран `onboarding_variant`. |
| `SetVariant(userID, variant)` | Валидирует `gb`/`us`, сохраняет, экран `onboarding_order`. |
| `SetOrder(userID, order)` | Валидирует `alpha`/`random`, сохраняет, экран `main_menu`. |
| `OpenMyWords(userID)` | Экран `my_words`. |
| `OpenMenu(userID)` | Экран `main_menu`. |

Невалидный ввод (неизвестный уровень и т. п.) → ошибка, состояние не меняется.

## 5. Поток Telegram (в `bot`)

1. **`/start`** → `service.Start` → отправить (`SendMessage`) рендер
   возвращённого экрана.
2. **callback `level:<v>` / `variant:<v>` / `order:<v>`** → соответствующий
   метод `service` → отредактировать сообщение (`EditMessageText`) рендером
   нового экрана → `AnswerCallbackQuery`.
3. **callback `menu:my_words`** → `service.OpenMyWords` → рендер экрана «Мои
   слова» (пусто) + кнопка «⬅️ Меню».
4. **callback `nav:menu`** → `service.OpenMenu` → рендер главного меню.

Callback-data самодостаточна (`вид:значение`), поэтому для кнопок чтение
текущего экрана не нужно. Чтение `state.screen` понадобится позже для
текстового ввода (Режим-2).

**Главное меню (минимум этапа):** одна кнопка **«📋 Мои слова»**.
**Экран «Мои слова»:** текст-заглушка
*«У вас пока нет слов в изучении. Скоро здесь появятся слова.»* + кнопка
**«⬅️ Меню»**.

## 6. Конфиг и запуск

- **Env:** `BOT_TOKEN` (обязателен), `MONGO_URI` (по умолчанию
  `mongodb://localhost:27017`), `MONGO_DB` (по умолчанию `irregular_verbs`).
- **`docker-compose.yml`:** сервис `mongo` + volume для локальной разработки.
- **Модуль:** `github.com/irbgeo/irregular-verbs-tgbot`; зависимости —
  `github.com/irbgeo/go-tgbot`, `go.mongodb.org/mongo-driver`.
- **Запуск:** `docker compose up -d` → задать `BOT_TOKEN` → `go run ./cmd/bot`.

## 7. Обработка ошибок

- **Telegram API:** ошибки логируются; при `retry_after` бот ждёт и повторяет;
  цикл `GetUpdates` при ошибке делает backoff и продолжает.
- **Mongo:** ошибки логируются; состояние FSM не теряется (в БД). Дружелюбное
  сообщение пользователю — отложено (см. §10).

## 8. Тестирование (TDD — тесты до кода)

- **`internal/service`:** загрузка всех 170 слов; сидинг (вызов `Upsert` на
  каждое слово через фейковый порт); переходы FSM (каждый use-case сохраняет
  нужную настройку и возвращает нужный экран); валидация ввода; идемпотентность
  `/start` для онбордившегося пользователя. Без сети и БД (фейковые порты).
- **`internal/bot`:** рендер экранов (контент кнопок, текст пустого «Мои
  слова»); роутер (реальный `service` + фейковый `UserRepository` + фейковый
  `Sender`) — апдейт приводит к нужному экрану.
- **`internal/store`:** тонкий интеграционный тест с Mongo, пропускается, если
  Mongo недоступна; проверка, что репозитории удовлетворяют портам.

## 9. Definition of Done

- Unit-тесты проходят.
- `docker compose up -d` поднимает Mongo.
- Ручной smoke-тест: `/start` → уровень → вариант → порядок → меню →
  «Мои слова» (пусто) → «⬅️ Меню».

## 10. Принятые допущения и отложенное

- Главное меню этапа — одна кнопка «Мои слова» (полное меню позже).
- Модуль: `github.com/irbgeo/irregular-verbs-tgbot`.
- Один инстанс бота (long polling).
- Карта `words` в `users` на этом этапе не создаётся.
- Дружелюбное сообщение пользователю при ошибке Mongo — отложено (ошибки
  логируются, состояние сохранено в БД).
- `Notifier` и `internal/reminder` — этап 5.
